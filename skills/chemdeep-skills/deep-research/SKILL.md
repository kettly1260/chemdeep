# 一键深度调研 (Deep Research)

ChemDeep 在 Cherry Studio / OpenClaw 中的**重型科研入口 skill**。当任务已经超过普通文献搜索，进入“需要完整研究框架、系统调研、机理路线比较、研究空白提炼、后续验证方向规划”的层级时，应优先进入本 skill，再由本 skill 决定何时调用 `run_deep_research`，而不是直接裸调单个重工具。

本 skill 调用 ChemDeep 的迭代式深度调研引擎，一键完成从文献检索到假设生成、评估和报告的全部流程。

## 默认入口定位

- 当用户明确要求“全面调研”“深度调研”“完整报告”“系统研究路线”时，**优先触发本 skill**
- 当任务已经明显超出普通检索、需要多轮整合分析时，**优先触发本 skill**
- 不要因为存在 `run_deep_research` 工具就直接绕过本 skill
- 本 skill 的职责是先确认研究目标、时间范围、质量阈值与 token/耗时预期，再决定是否执行重型调研
- 如果用户只需要普通搜索或分步分析，应回退到 `literature-survey`

## 适用场景

- 用户明确要求"全面调研"、"深度调研"或"完整报告"
- 需要从零开始对一个新课题进行系统性分析
- 用户愿意等待较长时间（通常 5-15 分钟）并接受较高 token 消耗

## 路由原则

1. **先确认，再执行重工具**：先进入本 skill，由本 skill 完成目标确认和成本提示，再决定是否调用 `run_deep_research`
2. **默认不要裸调重工具**：除非用户已经明确确认，否则不要直接调用 `run_deep_research`
3. **重型任务专用**：只有在需要完整研究框架与多轮综合分析时才使用本 skill
4. **轻型任务回退**：如果用户只是想找论文、筛论文或看单个问题，应引导回 `literature-survey`

## ⚠️ 重要：Token 消耗警告

`run_deep_research` 是一个重量级操作，它会在服务端自动执行多轮迭代，每轮包含：
- 文献检索和评分
- 证据提取
- 假设生成与评估
- 方法聚类
- 充分性检查

**预计消耗**：单次调用相当于 10-30 次普通工具调用的 token 量。

**必须在调用前向用户确认：**
1. 这是一个耗时较长的操作（5-15 分钟）
2. 会消耗较多 token
3. 确认研究目标是否准确

如果用户只需要简单的文献搜索或单个步骤，应引导其使用 `literature-survey`、`hypothesis-generation` 或 `verification-design` skill。

## 全文抓取与实时会话边界

`run_deep_research` 本身已经可以在需要时触发全文抓取流程，但对于受限出版商站点，是否能获取正文仍取决于用户当前浏览器里的合法授权会话。

当用户明确表示“希望尽量抓取全文”“希望连 PDF/出版社正文一起纳入调研”时：

1. 可先调用 `prepare_live_browser_session`
2. 提示用户在 Edge 中完成合法登录、验证码或 Cloudflare 验证
3. 必要时提示用户先手动打开目标文章页并点击一次 `View PDF`
4. 然后再执行 `run_deep_research`

**禁止错误表述：** 不要宣称该 skill 能绕过付费墙；它只能复用用户已拥有的访问权限。

## 工作流

### 步骤 1：确认研究目标

向用户确认：
- 研究方向和具体问题是什么？
- 有没有之前的调研基础或背景信息？
- 文献时间范围偏好？（默认近 5 年）
- 质量阈值要求？（默认评分 ≥5）

### 步骤 2：预览（可选但推荐）

建议先做一次快速的目标形式化，让用户确认方向：

```
→ formalize_research_goal(goal=<用户的研究目标>)
```

展示 ProblemSpec 给用户：
- "以下是我理解的研究框架，确认后将开始深度调研"
- 让用户有机会调整目标描述

### 步骤 3：确认执行

**必须**明确告知用户：

> ⚠️ 即将启动深度调研，这将：
> - 自动执行多轮文献检索和分析
> - 预计耗时 5-15 分钟
> - 消耗较多 token
>
> 确认继续？

**只有用户明确确认后才调用 `run_deep_research`。**

### 步骤 4：准备实时浏览器会话（可选）

仅当用户明确要求把出版社全文抓取能力纳入本次深度调研时执行：

```
→ prepare_live_browser_session(purpose="deep research full-text acquisition")
```

若工具返回需要人工操作，应先提示用户：
- 在浏览器里完成登录/验证
- 保持该窗口开启
- 然后再继续 `run_deep_research`

### 步骤 5：执行深度调研

```
→ run_deep_research(
    goal=<研究目标>,
    max_iterations=3,
    min_year=<时间范围>,
    min_score=<质量阈值>,
    previous_context=<之前的调研背景，如有>
  )
```

**参数建议**：
- `max_iterations`：默认 3，用户要求更深入可设为 5，不建议超过 5
- `min_year`：用户未指定时默认 2020
- `min_score`：用户未指定时默认 5.0

### 步骤 6：解读报告

收到结果后，将结构化数据转化为易读的报告：

1. **研究概述**
   - 形式化的研究目标
   - 检索策略和覆盖范围

2. **文献分析**
   - 高分论文数量和来源分布
   - 关键发现摘要

3. **机理假设**
   - 生成的假设列表
   - 各假设的证据评估状态
   - 最有前景的研究方向

4. **技术路径分析**
   - 方法聚类结果
   - 各路径的优劣对比

5. **研究空白与建议**
   - 当前未解决的问题
   - 建议的下一步研究方向

6. **全文获取情况**
   - 哪些论文成功获取了正文或 PDF
   - 哪些论文因权限、验证码或站点限制未能获取全文

7. **后续操作建议**
   - 如需设计验证方案，建议使用 `verification-design` skill
   - 如需补充检索，建议使用 `literature-survey` skill

## 约束

- **必须**在调用 `run_deep_research` 前获得用户明确确认
- **不要**将 `max_iterations` 设置超过 5
- **不要**在简单查询时使用此 skill，引导用户到合适的轻量级 skill
- **不要**在调研运行期间尝试调用其他 ChemDeep 工具（服务端可能会冲突）
- **不要**声称能够绕过出版社权限控制；全文抓取仅依赖用户合法授权会话
- 如果调研失败或超时，提供错误信息并建议用户改用分步 skill
- 报告输出应突出关键发现，不要简单复制原始 JSON 数据
